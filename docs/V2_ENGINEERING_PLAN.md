# iClassroom V2.0 Engineering Plan

本文档用于指导团队按工程化流程推进 V2.0。它不是产品 PRD，而是研发执行计划。

---

## 1. 目标

在现有 iClassroom MVP 基础上升级为 V2.0：

```text
管理者账号
→ 讲师账号
→ 讲师登录建活动
→ 观众扫码参与
→ 文字 / 图片 / 文件提交
→ 评分、看板、导出
```

核心策略：

- 不推倒重来。
- 保留现有可用主流程。
- 先补账号权限，再迁移老师接口。
- 新功能用小 PR 合并。

---

## 2. 阶段安排

### Phase 0：需求冻结与差距分析

周期：1 天

产出：

- V2.0 页面清单。
- V2.0 API 差距清单。
- V2.0 数据库差距清单。
- 旧 `teacherToken` 迁移策略。

验收：

- 团队明确 P0 / P1 / P2。
- 确认第一阶段不做选择题、加分项、词云、模板、举手提问。

### Phase 1：账号权限基础

周期：3-5 天

后端：

- 管理者账号。
- 讲师账号。
- 登录 / 登出 / 当前用户。
- 鉴权中间件。
- 讲师账号管理。

前端：

- 管理者登录。
- 讲师登录。
- 路由保护。
- 会话封装。

设计：

- 管理者端页面。
- 讲师登录和工作台框架。

验收：

- 管理者能创建讲师。
- 讲师能登录。
- 停用讲师无法登录。
- 未登录无法访问讲师后台。

### Phase 2：房间和观众入场

周期：3-5 天

后端：

- 房间关联讲师。
- 讲师只能访问自己的房间。
- 观众房间查询。
- 观众加入。
- 昵称验重。
- 小组满员。
- 断线恢复。

前端：

- 讲师创建房间。
- 观众扫码 / 输入房间号入场。
- 小组选择。

验收：

- 讲师登录后能建房。
- 观众无账号入场。
- 同房间昵称不能重复。
- 小组满员不可选。

### Phase 3：任务、提交、评分

周期：5-7 天

后端：

- 任务发布。
- 指定小组。
- 暂停 / 关闭 / 截止校验。
- 文字提交。
- 图片上传。
- 文件上传。
- 讲师审阅。
- 评分和小组分数事务更新。

前端：

- 任务管理。
- 观众答题页。
- 草稿自动保存。
- 图片 / 文件上传。
- 审阅评分页。

验收：

- 观众可提交文字、图片、文件。
- 上传失败保留文字草稿。
- 讲师可查看图片、下载文件。
- 评分只能 1-10。
- 小组分数正确累计。

### Phase 4：看板、归档、导出

周期：4-6 天

后端：

- 排行榜。
- 大屏数据。
- 精选答案。
- 后台数据。
- 活动结束。
- Excel + 原件 Zip 导出。

前端：

- 大屏看板。
- 后台数据看板。
- 精选答案操作。
- 导出和结束活动。

设计：

- AITIC 视觉走查。
- 大屏展示优化。
- 移动端适配走查。

验收：

- 大屏显示排行榜、完成度、精选答案。
- 后台显示参与率、得分对比、完成情况、提交时间分布。
- 活动结束后观众不能再进入。
- 导出包含 Excel、图片、文件。

### Phase 5：联调和验收

周期：2-4 天

必测链路：

```text
管理者创建讲师
→ 讲师登录
→ 讲师建房
→ 观众入场
→ 讲师发任务
→ 观众提交文字 / 图片 / 文件
→ 讲师评分
→ 大屏更新
→ 后台数据更新
→ 结束活动
→ 导出
```

## 3. 分支和 PR 规则

建议分支命名：

```text
feat/v2-auth
feat/v2-admin-teachers
feat/v2-room-ownership
feat/v2-submission-attachments
feat/v2-display-analytics
feat/v2-export
feat/v2-admin-frontend
feat/v2-teacher-frontend
feat/v2-student-upload
style/v2-aitic-theme
```

PR 原则：

- 一个 PR 只解决一个明确模块。
- PR 必须能独立说明和测试。
- 不允许一个 PR 同时改账号、上传、导出、大屏。
- 不允许混入无关格式化。

---

## 4. 验收清单

每个模块完成时必须回答：

- 改了哪些文件？
- 新增了哪些接口 / 页面 / 表？
- 是否影响旧功能？
- 如何本地运行？
- 如何测试？
- 有哪些风险？

---

## 5. 风险控制

| 风险 | 控制方式 |
| --- | --- |
| 旧 `teacherToken` 和新登录态冲突 | 先兼容字段，再逐步迁移接口 |
| 文件上传影响提交稳定性 | 先统一附件模型，再接前端上传 |
| 看板实时同步复杂 | 先 HTTP 拉取可用，再补 WebSocket 事件 |
| 权限绕过 | 所有 teacher/admin 接口统一中间件 |
| 大 PR 难 review | 严格按 Prompt 和模块拆分 |
