# iClassroom MVP 导师演示指南

## 0. 演示目标

本次展示的重点不是讲技术实现，而是演示一个老师真实上课时的互动流程。

核心表达：

> 这个版本是 iClassroom 的可演示 MVP，重点验证轻量课堂互动流程。老师和学生都不需要注册账号，老师创建课堂后分享房间码或链接，学生加入后完成任务提交，老师可以实时看到提交、批改反馈，并查看数据看板和课堂大屏。

## 1. 演示前准备

### 1.1 启动项目

如果使用 Docker：

```bash
cd /Users/lusitao/Downloads/iClassroom_AITIC/iClassroom
docker compose up --build
```

打开前端：

```text
http://localhost:3000
```

后端健康检查：

```text
http://localhost:8080/health
```

### 1.2 建议准备两个浏览器

用于模拟老师和学生：

- 老师端：Chrome 或 Edge
- 学生端：另一个浏览器、无痕窗口，或 Safari

说明：

> 本地演示时，每个浏览器会保存一个学生身份，所以可以用不同浏览器或无痕窗口模拟不同学生。

## 2. 推荐演示主线

演示顺序：

```text
首页
创建课堂
学生加入
老师发布任务
学生提交
老师批改
学生查看反馈
数据看板
大屏展示
说明二维码和文件上传下一阶段
```

## 3. 第一步：首页入口

打开：

```text
http://localhost:3000
```

展示内容：

- 页面标题：iClassroom 轻量课堂互动系统
- 老师端：创建课堂
- 学生端：输入房间码加入

讲解词：

> 这里是正式产品入口，不再是开发测试页面。老师可以创建课堂，学生可以输入房间码加入课堂。这个版本保持轻量级，不做账号注册登录，目标是让课堂互动尽快开始。

操作：

点击“创建课堂”。

## 4. 第二步：老师创建课堂

建议创建示例：

```text
课堂名称：产品设计课堂演示
小组数量：4
每组人数：5
```

讲解词：

> 老师创建课堂后，系统会生成唯一房间码。学生后续可以通过房间码或加入链接进入课堂。

操作重点：

- 创建成功后进入老师 dashboard
- URL 类似：`/teacher/rooms/:roomCode/dashboard`
- 记下页面中的房间码
- 展示顶部导航：概览、任务、批改、数据、大屏

## 5. 第三步：老师端概览

展示内容：

- 房间码
- 课堂状态
- 课堂信息
- 创建任务
- 导出数据
- 结束课堂

讲解词：

> 概览页是老师管理课堂的主入口，可以看到课堂状态、房间码、学生加入情况和任务情况，也可以结束课堂或导出数据。

注意：

不要一开始点“结束课堂”。这个可以最后说明，不要提前操作。

## 6. 第四步：学生加入课堂

打开另一个浏览器或无痕窗口：

```text
http://localhost:3000/student
```

操作：

1. 输入老师端显示的房间码
2. 点击“查找课堂”
3. 输入昵称
4. 选择小组
5. 点击加入课堂

建议学生昵称：

```text
小明
```

如果模拟第二个学生：

```text
小红
```

讲解词：

> 学生端不需要账号，输入老师提供的房间码即可加入。这个设计适合课堂现场快速互动，降低登录和注册成本。

## 7. 第五步：老师发布任务

回到老师端，点击顶部导航“任务”。

建议任务内容：

```text
任务标题：请用一句话描述你的小组方案
任务说明：请每位学生提交本组方案的核心亮点。
```

讲解词：

> 老师可以发布课堂任务，设置题目、说明、截止时间和面向对象。当前 MVP 主要验证文字任务提交流程。

操作：

- 创建任务
- 发布任务
- 回到学生端查看任务是否出现

## 8. 第六步：学生提交答案

回到学生端。

操作：

1. 进入任务
2. 输入答案
3. 提交

建议答案：

```text
我们小组的方案重点是降低老师课堂管理成本。
```

如果有第二个学生，可以提交：

```text
我们小组关注课堂现场的快速反馈和数据沉淀。
```

讲解词：

> 学生提交后，老师端可以看到提交记录。这个版本先支持文字提交，用来验证课堂互动主流程。

## 9. 第七步：老师批改

回到老师端，点击顶部导航“批改”。

操作：

1. 查看学生提交内容
2. 给学生打分
3. 填写反馈
4. 提交批改

建议反馈：

```text
表达清楚，可以继续补充具体使用场景。
```

讲解词：

> 老师可以查看学生提交内容，给分并写反馈。批改后，学生端可以看到老师给出的分数和评价。

## 10. 第八步：学生查看反馈

回到学生端。

展示内容：

- 学生提交结果
- 老师评分
- 老师反馈

讲解词：

> 到这里，已经完成了一个完整闭环：老师发布任务，学生提交，老师批改，学生收到反馈。

## 11. 第九步：数据看板

回到老师端，点击顶部导航“数据”。

展示内容：

- 课堂参与情况
- 提交情况
- 小组表现
- 数据统计

讲解词：

> 数据看板用于老师查看课堂参与情况、提交情况、小组表现和整体课堂数据。这是后续做课堂质量分析和教学反馈的基础。

## 12. 第十步：大屏展示

回到老师端，点击顶部导航“大屏”。

展示内容：

- 房间码
- 课堂大屏
- 小组排行
- 课堂展示信息

讲解词：

> 大屏适合投影到教室屏幕，用来展示房间码、课堂实时状态、小组排行和优秀答案。它的目标是增强课堂现场感。

二维码说明必须主动讲：

> 当前二维码是展示占位，不是真实可扫码二维码。原因是扫码必须依赖公网域名和 HTTPS。本地地址是 localhost，手机无法访问。部署到正式服务器后，二维码会绑定真实学生加入链接，学生扫码即可进入课堂。

## 13. 必须主动说明的未完成项

### 13.1 真实二维码扫码

讲解词：

> 真实二维码扫码目前没有接入，是因为它依赖正式公网地址。现在本地运行地址是 localhost，只能开发电脑访问，手机扫码无法打开。等部署到云服务器并配置域名和 HTTPS 后，接入二维码库即可生成真实可扫码二维码。

### 13.2 文件上传和 PDF

讲解词：

> 文件上传暂未放入本轮 MVP，是因为它不只是一个上传按钮，还涉及云存储、文件大小限制、类型校验、权限控制、预览、安全处理和批改链路。当前 MVP 先用文字提交验证核心互动流程，文件上传和 PDF 预览批注建议放到下一阶段。

## 14. 导师可能会问的问题

### Q1：为什么二维码现在不能扫？

回答：

> 因为当前是本地演示环境，二维码即使生成真实图片，也只能指向 localhost，手机无法访问。部署到正式服务器并配置域名 HTTPS 后，二维码会绑定公网学生加入链接，到时候就可以真实扫码。

### Q2：为什么现在不能上传 PDF？

回答：

> 文件上传涉及云存储、权限、安全校验、文件大小限制、在线预览和批改链路。为了保证 MVP 稳定，这一版优先验证课堂互动主流程。PDF 上传、预览和批注可以作为下一阶段 V1 功能。

### Q3：老师和学生为什么不用账号？

回答：

> 当前定位是轻量课堂互动，所以老师和学生都不需要注册登录。老师创建课堂，学生通过房间码加入。这样更适合课堂现场快速使用。后续如果要做长期课程管理，可以再扩展账号体系。

### Q4：上线后学生能不能用自己的手机加入？

回答：

> 可以。上线后只要有公网域名和 HTTPS，学生可以通过链接或二维码在自己的手机进入课堂。现在本地演示阶段主要受 localhost 限制。

### Q5：现在这个版本算完成了吗？

回答：

> 这个版本可以算可演示 MVP，已经跑通课堂互动主流程。正式上线前还需要补部署、真实二维码、HTTPS、生产数据库、基础安全和文件存储等能力。

## 15. 推荐收尾总结

可以这样收尾：

> 这个 MVP 当前已经能证明 iClassroom 的核心价值：老师可以低门槛创建课堂，学生可以快速加入互动，老师可以发布任务、查看提交、批改反馈，并通过数据看板和大屏看到课堂状态。下一阶段建议优先补真实二维码、复制加入链接、线上部署和文件上传能力。

## 16. 演示提醒

- 不要提前点“结束课堂”
- 房间码一定要记下来
- 学生端建议用另一个浏览器或无痕窗口
- 如果学生端没有立刻看到任务，可以刷新页面
- 二维码和文件上传要主动说明，不要等导师误以为功能坏了
- 演示重点是完整闭环，不要在单个页面停留太久
