# iClassroom V2.0 Codex / Codex Engineering Guide

本文件是 iClassroom V2.0 的 AI 协作最高规则。Codex、Codex、Cursor 等工具在本项目中生成代码、文档或测试前，必须先阅读本文件，再阅读对应任务文档。

V2.0 不再是“无账号课堂 MVP”，而是面向展厅 / 体验中心运营的轻量网页互动平台。

---

## 0. 项目目标

开发一个面向展厅运营场景的纯网页互动平台：

```text
管理者配置讲师账号
→ 讲师登录
→ 讲师创建活动房间
→ 观众扫码 / 链接入场
→ 观众填写昵称并选择小组
→ 讲师发布任务
→ 观众提交文字 / 图片 / 文件
→ 讲师按小组审阅评分
→ 排行榜、大屏、后台数据实时更新
→ 活动结束归档
→ 导出 Excel + 原件 Zip
```

核心原则：

```text
先账号权限，再主流程；
先 HTTP 契约，再实时同步；
先业务正确性，再视觉增强；
先小 PR 验收，再跨模块联调。
```

## 2. V2.0 角色与权限

### 2.1 管理者 Admin

- 有账号，电脑网页端使用。
- 可创建、停用、删除、重置讲师账号。
- 可查看 P1 运营概览。
- 不直接参与单场活动评分，除非另有讲师身份。

### 2.2 讲师 Teacher

- 有账号，由管理者分配。
- 必须登录后才能创建房间、发布任务、评分、看板、导出。
- 只能访问和管理自己创建的活动。
- 不能管理其他讲师账号。

### 2.3 观众 Audience

- 无账号。
- 通过房间号、二维码或链接入场。
- 使用昵称、小组和 `clientToken` 标识本次活动身份。
- 活动结束后不能再次进入操作。

---

## 3. 身份与会话规则

V2.0 的身份规则如下：

```text
管理者端：账号密码登录，使用管理员会话访问 admin 接口
讲师端：账号密码登录，使用讲师会话访问 teacher 接口
观众端：无账号，使用 roomCode + nickname + group + clientToken
```

旧 MVP 的 `teacherToken` 只能作为迁移兼容字段，不得继续作为 V2.0 主要讲师鉴权方案。

所有新增管理者 / 讲师接口必须通过统一鉴权中间件，不允许只依赖 `roomCode`、前端隐藏按钮或本地状态判断权限。

## 4. 开发行为准则

AI 在写代码前必须：

1. 检查当前目录结构。
2. 检查已有代码，不重复创建同名模块。
3. 明确本次任务属于哪个 Prompt / 哪个模块。
4. 只开发当前模块，不跨阶段生成大量功能。
5. 说明会修改哪些文件。

AI 完成任务后必须输出：

1. 修改文件清单。
2. 新增接口、页面或数据表。
3. 本地运行方式。
4. 测试方式和测试结果。
5. 需要人工检查的点。
6. 已知风险。

禁止：

- 提交 `.env`、密钥、token、数据库密码。
- 为了跑通而删除权限校验、状态校验或业务约束。
- handler 直接写复杂 SQL。
- handler 直接操作多个表完成业务。
- 前端页面直接硬编码大量接口 URL。
- 在一个 PR 内同时改账号、上传、看板、导出等多个主模块。
- 未说明原因就重写已有架构。

## 5. 后端分层规则

推荐结构：

```text
backend/
├── cmd/server/main.go
├── internal/
│   ├── config/
│   ├── domain/
│   ├── repository/
│   ├── service/
│   ├── handler/
│   ├── middleware/
│   ├── websocket/
│   ├── storage/
│   ├── export/
│   ├── auth/
│   └── response/
├── migrations/
├── uploads/
└── .env.example
```

职责边界：

```text
handler：HTTP 参数绑定、调用 service、返回 response
service：业务规则、事务、状态流转、权限后的业务校验
repository：数据库读写
domain：实体、枚举、领域对象
middleware/auth：登录态解析、角色校验、接口鉴权
storage：图片和文件存储
export：Excel 和 Zip 导出
websocket：实时连接和事件广播
```

## 6. 前端结构规则

推荐结构：

```text
frontend/src/
├── api/
├── router/
├── pages/
│   ├── admin/
│   ├── teacher/
│   ├── student/
│   └── display/
├── components/
├── hooks/
├── utils/
├── types/
└── styles/
```

前端必须做到：

- API 请求集中封装在 `api/`。
- 管理者、讲师、观众路由边界清晰。
- 登录态和本地身份读写封装在 `utils/session.ts` 或同类模块。
- 讲师端和管理者端电脑端优先。
- 观众端手机优先，同时响应式覆盖平板和电脑。
- 大屏端全屏展示优先。
- 新视觉遵守 AITIC 配色：阿里橙 + 深墨 + 暖中性。

## 7. 统一业务规则

### 7.1 账号规则

- 管理者可创建、停用、删除、重置讲师账号。
- 停用后的讲师不能登录。
- 停用不删除历史活动数据。
- 讲师账号不能自助注册。

### 7.2 房间规则

- 房间由已登录讲师创建。
- 房间必须关联 `teacherId`。
- 系统生成唯一 `roomCode`。
- 状态为 `created`、`active`、`ended`。
- 活动结束后观众不能加入、恢复、提交。
- 讲师仍可查看和导出自己活动的数据。

### 7.3 观众规则

- 观众无需注册。
- 同一房间内昵称唯一。
- 小组满员不可加入。
- 后端生成 `clientToken`，用于断线重连和后续观众接口。

### 7.4 任务规则

- 讲师发布任务，可推送全场或指定小组。
- 任务包含标题、描述、可选附件、截止时间。
- 状态为 `published`、`paused`、`closed`。
- 暂停、关闭或超过截止时间后不能提交。

### 7.5 提交规则

- 一名观众对一个任务只能提交一次。
- 提交支持文字、图片、文件。
- 图片单图不超过 5MB，单题最多 3 张。
- 文件不含视频，类型和大小上限以 `docs/api.md` 为准。
- 上传失败时前端必须保留文字草稿。

### 7.6 评分规则

- 讲师手动评分，整数 1-10，最低 1 分。
- 可填写评语。
- 保存评分后提交状态变为 `graded`。
- 小组总分按评分差值事务更新。

### 7.7 看板与导出规则

- 大屏展示排行榜、完成度、精选答案。
- 后台数据看板展示参与率、得分对比、任务完成情况、提交时间分布。
- 导出包含 Excel + 图片 / 文件原件 Zip。
- 图片 / 文件线上保留 30 天，保留期内可导出。

## 8. V2.0 开发顺序

必须按以下顺序推进：

1. 项目审计与差距分析。
2. 数据库模型改造。
3. 管理者 / 讲师账号与鉴权。
4. 房间接口从旧 `teacherToken` 模式迁移到讲师账号模式。
5. 观众入场与匿名会话。
6. 任务发布与截止控制。
7. 文字 / 图片 / 文件提交。
8. 讲师审阅与评分。
9. 排行榜、大屏、精选答案。
10. 后台数据看板。
11. 活动结束、归档、导出。
12. 前端管理者端。
13. 前端讲师端账号化改造。
14. 前端观众端上传与移动端体验。
15. AITIC 视觉体系改造。
16. E2E 测试和验收。

## 9. Prompt 使用规则

Prompt 必须小而明确：

```text
一个阶段
一个模块
一个 Prompt
一个 PR
```

禁止把完整 V2.0 需求一次性发给 AI 让其生成全项目。

每个 Prompt 必须包含：

- 本次目标。
- 必读文档。
- 允许修改文件。
- 禁止修改文件。
- 接口 / 页面 / 数据库验收标准。
- 必须运行的测试。
- 完成后需要输出的清单。

具体 Prompt 模板见 `docs/V2_PROMPT_PACK.md`。

## 10. 测试要求

后端：

- 修改 service / repository 必须补单元测试。
- 修改 handler 必须补接口级测试或清晰手测步骤。
- 每个后端 PR 默认运行 `go test ./...`。

前端：

- 修改 API client 必须确认调用路径、Header、错误处理。
- 修改路由必须确认未登录、无权限、会话过期场景。
- 修改观众端必须检查手机尺寸。
- 修改上传必须检查失败、超限、重试、草稿保留。

联调：

- 管理者创建讲师。
- 讲师登录建房。
- 观众入场。
- 讲师发任务。
- 观众提交文字 / 图片 / 文件。
- 讲师评分。
- 大屏和后台数据更新。
- 结束活动后观众无法进入。
- 导出 Excel + 原件 Zip。
